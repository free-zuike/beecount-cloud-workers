# Fix: Charts show no data + TypeScript build error

## Root Cause Analysis

### Bug 1: Charts (MonthlyTrendBars, HomeYearHeatmap) show no data
**File**: `src/routes/workspace.ts:844-846`

The analytics endpoint filters by `external_id` when `ledger_id` is provided:
```ts
if (ledgerId) {
    ledgerQuery += ' AND external_id = ?';
    ledgerParams.push(ledgerId);
}
```

The web frontend sends `ledger_id=9ee4d2c7-d685-4e58-be56-bd6196652050` (the ledger's **internal** `id`, not `external_id`). The SQL query `WHERE external_id = '9ee4d2c7-...'` finds no matching ledgers â†’ `ledgers.results.length === 0` â†’ returns empty `{ series: [], ... }`.

The hero card works because it uses `currentMonthSummary` (from `rMonthly` call at index [2]) and `ledgerCounts` (from `rCounts` at index [4]), which may not filter by `ledger_id` the same way, or the summary endpoint was already fixed.

**Fix**: Change `AND external_id = ?` to `AND (external_id = ? OR l.id = ?)` â€” same pattern used in other fixed endpoints.

### Bug 2: TypeScript build error blocks CI deployment
**File**: `frontend/apps/web/src/context/SyncSocketContext.tsx:235-236`

```ts
const p = payload as SyncErrorPayload
handlerRef.current(p.errors)
```

`SyncEventBase` has `[key: string]: unknown` index signature, which makes `p.errors` resolve to `unknown` even after `as SyncErrorPayload` cast (TypeScript 5.x behavior with union types + index signatures).

This error **prevents the frontend from building in CI**, meaning the latest code (including backend fixes) never gets deployed as a new frontend bundle.

**Fix**: Add explicit type assertion on `.errors`, or restructure to avoid the index signature conflict.

## Changes

### 1. Fix analytics endpoint ledger_id query (`src/routes/workspace.ts`)
**Line 844-846**: Change:
```ts
if (ledgerId) {
    ledgerQuery += ' AND external_id = ?';
    ledgerParams.push(ledgerId);
}
```
To:
```ts
if (ledgerId) {
    ledgerQuery += ' AND (external_id = ? OR l.id = ?)';
    ledgerParams.push(ledgerId, ledgerId);
}
```
This tries both `external_id` and internal `id`, same pattern as `read.ts` and `summary.ts` fixes.

### 2. Fix TypeScript build error (`frontend/apps/web/src/context/SyncSocketContext.tsx`)
**Lines 233-236**: Change:
```ts
const wrapped = useCallback((payload: SyncEventPayload) => {
    if (payload.type === 'sync_error') {
      const p = payload as SyncErrorPayload
      handlerRef.current(p.errors)
    }
  }, [])
```
To:
```ts
const wrapped = useCallback((payload: SyncEventPayload) => {
    if (payload.type === 'sync_error') {
      const p = payload as unknown as SyncErrorPayload
      handlerRef.current(p.errors)
    }
  }, [])
```
Or alternatively, remove the intermediate variable:
```ts
const wrapped = useCallback((payload: SyncEventPayload) => {
    if (payload.type === 'sync_error') {
      handlerRef.current((payload as SyncErrorPayload).errors)
    }
  }, [])
```
